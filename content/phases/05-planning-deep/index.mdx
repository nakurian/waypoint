# Phase 05 — Planning (deep)

You got the Go decision in [Phase 04](/phase/04). Now figure out exactly what you are building. The output of this phase is a complete set of requirements — functional, non-functional, security, integration — and a stack of trackable stories with acceptance criteria good enough that `/ticket-to-pr` will accept them without argument.

[Phase 02](/phase/02) is the short version of this page aimed at a single ticket. Phase 05 is the long version aimed at a body of work: an initiative, a program increment, or a quarter's worth of epics.

## Goal of this phase

Complete requirements documentation with stakeholder sign-off, defining **WHAT** the system must do and **HOW WELL** it must perform. "How" is the next phase's problem.

## The six steps of deep planning

### 1. Gather stakeholder input

Find everyone who will have an opinion after the fact and get the opinion before the fact. That includes the obvious — product, the asking business unit, the lead of every team whose code will change — and the non-obvious: support, security, operations, customer success. Meeting summaries belong in the team's docs location; action items belong as tracker tickets with owners.

<Callout type="note">
Waypoint v1.5 will ship `/meeting-processor` to convert meeting notes into structured decisions, action items, and requirements. In v1 this step is manual; the discipline below still applies.
</Callout>

### 2. Define scope boundaries

Scope is a two-column list. In-scope on the left, out-of-scope on the right. Short phrases; no hedging. The out-of-scope column is the more important one — it is where you say "not this, not now, not here" while everyone in the room can still hear you.

If you find yourself writing "in scope: everything necessary to support X," rewrite it. That is not scope, it is a promise.

### 3. Document functional requirements

Functional requirements say what the system does. They are the raw material from which stories are cut. Each requirement:

- Names the actor, the trigger, and the expected outcome
- Is traceable to a specific point in scope (every requirement has a reason to exist)
- Is atomic — a requirement that contains the word "and" is probably two requirements

From the functional requirements you generate tracker stories, each with **SMART** acceptance criteria (Specific, Measurable, Achievable, Relevant, Time-bounded) in the *Given X, when Y, then Z* form [Phase 02](/phase/02) describes. Waypoint's `/ticket-to-pr` skill will refuse to proceed on stories whose ACs fail this shape, so shape them right the first time.

<Callout type="success">
A functional requirement: "The system allows an authenticated account holder to download their last 12 months of transactions as CSV." → Story with ACs on export format, date-range controls, authentication failure modes, and file-size limits.
</Callout>

### 4. Define non-functional requirements

Functional requirements say what the system does. Non-functional requirements (NFRs) say how well. This is where most requirements sets fall apart — "fast" is not a requirement, "< 200ms p95 under 500 RPS" is.

The NFR categories Waypoint recommends you fill in explicitly:

| Category | Example NFR |
|---|---|
| Performance | p95 latency < 200ms for the core read path at 500 RPS sustained |
| Availability | 99.9% monthly uptime measured as successful-response ratio on `/health` |
| Scalability | Horizontally scales to 10x current load without architectural change |
| Security | All data at rest encrypted with AES-256; all data in transit TLS 1.3+ |
| Observability | p50/p95/p99 on every public endpoint; structured logs with correlation IDs |
| Accessibility | WCAG 2.1 AA on every user-facing surface |
| Compliance | Any regulatory obligations — PCI DSS, SOC 2, GDPR, HIPAA — named explicitly |

Every NFR should be measurable. If you cannot write a test for it, it is an aspiration, not a requirement.

### 5. Identify integration requirements

Map every external system the change will touch. For each integration:

- Name and owner (internal team or vendor)
- Synchronous or asynchronous
- Expected latency and availability contract
- Authentication and authorisation model
- Failure mode — what happens when the other side is down?
- Data contract: what goes in, what comes out, in what shape

Waypoint v1.5 will ship `/api-documentation` and `/mermaid-diagrams` skills for this step. In v1 you compose the integration spec by hand. Use the Context7 MCP server to fetch current docs for any library or API you are integrating against — the model's training-data knowledge may be stale.

### 6. Review and sign-off

A sign-off meeting with all stakeholders from step 1. Walk the room through scope, functional requirements, NFRs, integrations. Capture any changes as version-controlled edits — every adjustment has an author, a timestamp, and a reason. Record the approval explicitly: who signed off, what was approved, on what date.

An unsigned requirements document is not a plan, it is a draft.

## Deliverables

At the end of Phase 05 you should have:

- Stakeholder input summary
- Scope definition (in / out)
- Functional requirements, with tracker stories generated from them
- Non-functional requirements with measurable targets
- Security requirements (a subset of NFRs that deserve their own section)
- Integration specifications
- A signed-off record of approval

How you physically store these — Confluence, a docs repo, a wiki — is your team's choice. Waypoint has no opinion beyond "version-controlled and findable."

## Exit criteria

**Must have:**
- Scope boundaries clearly defined
- All functional requirements have acceptance criteria in SMART form
- Non-functional requirements have measurable targets
- External integrations identified with contracts
- Stakeholder sign-off obtained

**Should have:**
- Traceability established — each story traces to a requirement, each requirement to the business case
- User stories pass the INVEST test (Independent, Negotiable, Valuable, Estimable, Small, Testable)
- All assumptions explicitly documented

## Common pitfalls

| Avoid | Do instead |
|---|---|
| Mixing requirements with design | Keep HOW out; focus on WHAT |
| Vague acceptance criteria | Make everything measurable and testable |
| Skipping NFRs | "Fast" is not a requirement; "< 200ms p95" is |
| Gold-plating (too many features) | Prioritise with MoSCoW; ship the Must-haves first |
| Analysis paralysis | Reach 80% and iterate — a perfect doc that takes a quarter to produce has already missed its window |
| Writing requirements the model will auto-complete | Requirements are for humans first; `/ticket-to-pr` reads them as input, not as its own author |

## What this looks like in practice

A lead on an online-travel-agency team is turning the refund-workflow initiative from [Phase 04](/phase/04) into requirements.

- **Scope** includes: parallelising the three backend calls, adding a batch variant to the third API, new client-side caching. **Out of scope:** redesigning the agent UI, refactoring the refund state machine, mobile support.
- **Functional requirements** cover: the agent can look up a refund in under 10 seconds (SLA), the result is consistent across concurrent lookups, audit trail is preserved. Each becomes 2-4 tracker stories.
- **NFRs** add: p95 latency < 8 seconds at current peak load; 99.95% availability during business hours; all refund access logged with agent ID and timestamp.
- **Integrations** map the three backends, note their current p95s, specify the new batch endpoint's contract, and identify the vendor-dependency risk on the third one.
- **Sign-off** is a 45-minute meeting with the agent-tools product owner, the three backend leads, security, and the call-centre operations lead. Recorded approval, no open questions.

The stories produced here are what `/ticket-to-pr` will consume in [Phase 07](/phase/07). The architecture questions raised — caching strategy, retry semantics, the parallel-call coordination — are the feed for [Phase 06](/phase/06).
